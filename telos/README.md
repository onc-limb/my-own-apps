# telos

> ビジネスパーソンの「散らかった仕事の情報」を、判断と行動につながる形に整理する web アプリ。
> τέλος（テロス）= ギリシャ語で「目的・最終目標」。すべての整理は目的のためにある。

## なぜ作るのか

仕事の情報は放っておくと3つの形で散らかる。

1. **やるべきことが頭の中にだけある** — リリース前の確認事項を記憶と経験に頼ると、必ず抜ける。外科手術ですらチェックリストの導入で死亡率が半減した（Haynes et al., 2009）。ソフトウェアリリースも同じ構造の問題を抱えている。
2. **報告が「起きたことの羅列」になる** — 進捗報告は、聞き手（上司・経営層）が知りたい「結論」から始まらないと伝わらない。結論を先に置く構造（ピラミッド原則）は、意思決定者の認知の流れに合っている。
3. **会話が流れて、本当の目的が埋もれる** — 打ち合わせやチャットでは要望・不満が大量に出るが、その奥にある「本当に叶えたいこと」は明示されない。表層の要望のまま作ると、誰も使わないものができる。

詳細なエビデンスは [docs/concept.md](docs/concept.md)。

## コアコンセプト

> **情報を「記録」で終わらせず、目的（telos）に向かう道具に変える。**

10のツールは全部この軸から導かれ、案件のライフサイクルに沿って並ぶ：

**入口を勝つ**
- **Positioning** — 「自分の強み」を棚卸しし、提供価値を一文に言語化する
- **Proposal** — 「ふわっとした相談」を受注につながる提案書に変える
- **Estimate** — 「曖昧な依頼」を前提・含むこと・含まないことに分けて見積る

**案件を進める**
- **Goals** — 「言われたこと」を掘り下げ、本当の目的を見つける
- **Decisions** — 「議事」から決定・未決・宿題を抜き出し、認識を揃える
- **Changes** — 「スコープ外の追加依頼」を、影響つきの見える変更にする
- **Checklist** — 「やるべきこと」を記憶から外に出し、リリース判断の道具にする

**報告・説明する**
- **Slides** — 「起きたこと」を結論ファーストに組み替え、報告の道具にする
- **Brief** — 「決めてほしいこと」を一枚に整理し、ビジネス側へ説明する

**次につなげる**
- **Retrospective** — 「終わった案件」を振り返り、次フェーズの継続提案に変える

## 構成（pnpm ワークスペース）

- `app/` — Vite + TypeScript（フレームワークなし、クライアントサイドのみ、localStorage 永続化）
- `server/` — **任意** の LLM ブリッジ。`claude -p`（サブスクリプション）または LiteLLM 互換 API（API キー）で AI 支援を有効化する

```bash
pnpm install
pnpm dev        # app を起動（http://localhost:5173）

# AI 支援を使う場合（任意）
pnpm bridge     # LLM ブリッジを起動（http://localhost:8787）
```

AI を使わなくても全ツールはヒューリスティックで動作する。AI 連携・データのインポート/エクスポートは
app の **Settings** 画面から。ブリッジの詳細は [server/README.md](server/README.md)。
