# telos — 機能一覧

すべての機能は「情報を記録で終わらせず、目的に向かう道具に変える」というコンセプトから導かれる。

## Phase 1 — MVP + α（実装済み）

### Checklist — リリースチェックリスト

| 機能 | 説明 |
|------|------|
| テンプレートから作成 | 7カテゴリ（企画・要件 / 開発・品質 / セキュリティ / 運用準備 / ドキュメント / 告知・サポート / リリース後）約25項目の標準リリースチェックリストから新規作成 |
| 項目のチェック / 進捗表示 | カテゴリ別 + 全体の進捗バー。リリース判定の状態が一目で分かる |
| 項目の追加・削除 | プロジェクト固有の確認事項を追加できる |
| 複数チェックリスト管理 | リリースごとに独立したインスタンスを保持 |

### Slides — 進捗報告スライド

| 機能 | 説明 |
|------|------|
| フォーム入力 → スライド生成 | プロジェクト名・期間・状況（順調/注意/危険）・サマリー・成果・課題・次アクションを入力すると、結論ファースト構造のスライドが生成される |
| プレゼンモード | スライドを1枚ずつ表示。キーボード（←/→）で操作 |
| PDF 出力 | ブラウザ印刷で1スライド=1ページの PDF に出力（印刷 CSS） |
| 報告書の保存・複製 | 過去の報告を保存し、次回の報告のベースにできる |

### Goals — 目的の発見

| 機能 | 説明 |
|------|------|
| 会話ログの貼り付け・分析 | 議事録・チャットログを貼り付けると、要望・不満・目的を示す発言を自動抽出（「〜したい」「〜が課題」等のマーカー検出）し、頻出キーワードを表示 |
| 5 Whys ガイド | 抽出された発言から1つ選び、「なぜそれが重要か」を最大5段階掘り下げるガイド付きフォーム |
| JTBD 形式の目的文 | 掘り下げの結果を「（状況）のとき、（動機）したい。そうすれば（期待する進歩）」の形式に着地させる |
| セッション保存 | 分析と掘り下げの過程を保存し、後から見返せる |

### Brief — ビジネス説明資料

| 機能 | 説明 |
|------|------|
| 一枚資料の作成 | 背景・課題 / 提案 / 期待効果 / コスト・前提 / 意思決定のお願い の構成で、意思決定者が30秒で読めるクリーンな一枚資料を作成 |
| クリーンな出力 | アプリ本体はダークなエンジニア系 UI、成果物（資料）は余白の効いた白い「紙」として表示。ごちゃごちゃさせない |
| AI 下書き生成（任意） | 箇条書きメモを貼ると、LLM が各セクションに整理して下書きを生成 |
| PDF 出力 | ブラウザ印刷で A4 一枚の PDF に出力 |

### Positioning — 自分の価値の言語化

フリーランスエンジニアが案件の入口で勝つための土台。「何屋さんですか?」に一文で答えられる状態をつくる。

| 機能 | 説明 |
|------|------|
| 強みの棚卸し → JTBD 言語化 | 強み・実績メモから「誰の・どんな進歩を・どう実現するか」を整理し、価値の一文を組み立てる |
| バリューステートメント出力 | プロフィール・営業文・提案の冒頭に貼れるクリーンな価値カード（白い紙）を生成・PDF 出力 |
| AI 言語化（任意） | 強みメモから target / job / approach / 一文 / 実績を LLM が提案 |

### Retrospective — 振り返り→継続提案

リピート・紹介につなげる出口の道具。納品で終わらせず、振り返りを次フェーズの提案に変える。

| 機能 | 説明 |
|------|------|
| 振り返りの構造化 | 成果・続けたいこと（Keep）・残った課題（Problem）を整理 |
| 継続提案の生成 | 振り返りから「今回の成果 → 残課題 → 次フェーズで実現すること」の継続提案文をテンプレ or AI で生成 |
| クライアント向け一枚出力 | 振り返り + 次のご提案を白い紙にまとめ、PDF 出力 |
| AI 提案生成（任意） | 残課題から次にできること（継続提案の種）と提案文面を LLM が生成 |

### Estimate — 見積・スコープ定義

受託の赤字は「スコープ膨張による実質値下げ」から始まる。それを入口で塞ぐ。

| 機能 | 説明 |
|------|------|
| 作業分解と工数レンジ | 作業項目ごとに min/max の人日を入力。合計工数レンジと（単価を入れれば）概算金額レンジを自動計算 |
| 前提・含まないことの明示 | 「本見積に含まないもの」を独立セクションで明示し、後の追加依頼との線引きにする |
| 見積書出力 | クライアントに渡せる見積書（白い紙）として PDF 出力 |
| AI 作業分解（任意） | 依頼メモから作業項目・工数レンジ・前提・含まないことを LLM が分解 |

### Proposal — 提案書

受注はここで決まる。Goals で掘った目的・Positioning の価値・Estimate の概算を一本の提案書に束ねる。

| 機能 | 説明 |
|------|------|
| 提案書の構成 | 課題理解 → アプローチ → 進め方（フェーズ）→ 概算 → なぜ自分か、の構成を固定 |
| 提案書出力 | クライアントに渡せる提案書（白い紙）として PDF 出力 |
| AI 下書き（任意） | 案件メモから各セクションの下書きを LLM が生成 |

### Decisions — 議事録→決定事項

非エンジニアのクライアントとの「言った言わない」を防ぎ、プロ感で信頼を得る。

| 機能 | 説明 |
|------|------|
| 決定・未決・アクションの抽出 | 打ち合わせログから、決定事項 / 未決・確認事項 / アクション（担当付き）をマーカー検出で自動分類 |
| 手直し | 抽出結果をそのまま編集できる。アクションは「担当: 内容」形式 |
| 議事録出力 | 関係者に共有できる議事録（白い紙）として PDF 出力 |
| AI 抽出（任意） | LLM が決定・未決・アクションを精度高く切り分け |

### Changes — スコープ変更管理

「ついでにこれも」を、追加請求できる見える変更に変える。エンジニアが最も取りこぼす収益源を守る。

| 機能 | 説明 |
|------|------|
| 変更依頼の記録 | 追加依頼ごとに、背景・追加工数（人日）・請求対象か・状態（提案中/合意/却下）を管理 |
| 取りこぼし防止の集計 | 「合意済み・請求対象」の追加分だけを集計し、追加工数・概算金額を表示 |
| 変更管理表出力 | クライアントと合意するための変更一覧（白い紙）として PDF 出力 |

### 共通機能

| 機能 | 説明 |
|------|------|
| データのインポート / エクスポート | 全データ（チェックリスト・報告・目的・資料）を JSON でバックアップ / 復元。端末間移行に対応。取り込みは「追加」「置き換え」を選べる |
| LLM 連携（任意） | ローカルブリッジ経由で AI 支援。**サブスクリプション（`claude -p`）** と **API キー（LiteLLM 互換）** の両方に対応。ブラウザに API キーを持たせない設計 |
