# 企画資料テンプレート

`plan-product` の Phase 5 で生成する企画資料の雛形。`{app}` はアプリ名（ディレクトリ名）に置き換える。実物の良い例は `marginalia/`（初適用例）を参照。

各ファイルは「全機能を1本の軸（コンセプト）から導く」という方針を共有する。コンセプトは Phase 2 で一文に磨いたものを使う。

---

## {app}/README.md

アプリの顔。一目で「何を・なぜ・どう違うか」が分かること。

```markdown
# {app}

> 一言でアプリを説明する（誰の何を解決するか）。

## なぜ作るのか

既存手段（例: Kindle）への不満と、その根っこにある動機を述べる。
可能なら「これは認知科学／研究でも裏付けられている」と一言（詳細は docs/concept.md へリンク）。

## コアコンセプト

> **一文で言い切るコンセプト（Phase 2 の成果）。**

その狙いを 1〜2 文で補足。

## 差別化の核

- **機能A**: コンセプトをどう体現するか。
- **機能B**: 〃
- **設計上の判断C**: 〃

## ドキュメント

- [docs/concept.md](docs/concept.md) — 差別化コンセプトとエビデンス
- [docs/features.md](docs/features.md) — 機能一覧
- [roadmap/README.md](roadmap/README.md) — ロードマップ

## ステータス

企画段階／実装中など。アイデア出しの記録は journal を参照。
```

---

## {app}/docs/concept.md

差別化コンセプトの本体。**問題 → エビデンス → 知見と機能の対応 → 出典** の順で書くと説得力が出る。

```markdown
# 差別化コンセプト — {一文コンセプト}

## 問題: {根っこにある一つの問題}

ユーザーが感じる不満（Phase 1 で集めたもの）を箇条書き。
それらが「個別の不満ではなく、根っこは一つ＝○○」だと示す。

## エビデンス

### {知見の見出し1}
研究・メタ分析の知見を、著者・年とともに要約。

### {知見の見出し2}
〃

## {app} の解決アプローチ（知見と機能の対応）

| 知見 | {app} の機能 |
|------|-------------|
| 知見1 | 機能X |
| 知見2 | 機能Y |

## 出典

- 著者 (年). *タイトル.* 媒体. <URL>
```

## {app}/docs/features.md

機能一覧。優先度はフェーズで表現し、各機能がコンセプトの軸に乗ることを意識する。

```markdown
# 機能一覧

優先度はフェーズで表現する（roadmap 参照）。すべての機能は「{一文コンセプト}」という軸から導かれる。

## コア: {柱の名前}（MVP）
- ...

## {領域}（MVP）
- ...

## {領域}（フェーズ2）
- ...

## 発展（フェーズ3）
- ...
```

## {app}/roadmap/README.md

ロードマップの入口。設計哲学・（決まっていれば技術スタック）・フェーズ概要・MVP の位置づけ。

```markdown
# {app} ロードマップ

> コンセプト: {一文}。詳細は ../docs/concept.md、機能は ../docs/features.md。

## 設計哲学
- 機能を盛るのではなく、1本の軸から全機能を導く。
- （プロダクト固有の原則）

## 技術スタック（決定したら）
| レイヤー | 採用 | 補足 |
|----------|------|------|
| 言語・UI | ... | ... |

## フェーズ概要
| Phase | テーマ | ゴール |
|-------|--------|--------|
| 1 (MVP) | ... | ... |
| 2 | ... | ... |
| 3 | ... | ... |

## Phase 1 (MVP)
ゴールとスコープ判断（何を入れ、何を後回しにするか）。詳細は phase1-*.md。
```

## {app}/roadmap/phaseN-*.md（任意 / 実装計画を詰めるとき）

実装に入る段で、フェーズの詳細を落とす。

```markdown
# Phase N — {テーマ}

> ゴール: {このフェーズで到達する状態}。

## スコープ
対象と非対象。実装難度の高い要素を後回しにする判断を明記。

## 技術
使うフレームワーク・ライブラリ。

## 実装ステップ
各ステップで動作確認できる「垂直スライス」で、上から積み上げる。
### Step 1: ...
- やること
- **確認:** どうなれば完了か

## データモデル
永続化するエンティティと主要フィールド。

## 完成基準
- [ ] チェックリスト
```
