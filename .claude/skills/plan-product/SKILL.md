# Plan Product — アイデア出しから企画資料まで

新しい個人アプリの構想を、アイデアの壁打ちから企画資料の作成まで一気通貫で支援する。`my-own-apps` における Claude の役割（アイデア出しの壁打ち、やり取りの記録、skill 化）を体系化したもの。

## 基本姿勢（なぜこの順序か）

背骨は **Double Diamond**：問題空間（なぜ・何が不満か）を十分に広げてから1つに絞り、次に解決空間（どう作るか）を広げてから絞る。最初から解決策に飛びつかない。

理由：良いプロダクトは「機能」ではなく「ユーザーが遂げたい進歩（**Jobs to Be Done**）」から生まれる。動機を取り違えると、立派でも誰も使わない機能を作ってしまう。だから Phase 1〜2 で「何が・なぜ嫌か」を徹底的に掘り、コンセプト（=その人が本当に求めている進歩）に収束させてから、解決策の話に進む。

## フェーズ

### Phase 1 — 動機を深掘る（問題空間を発散）

目的：ユーザーが本当に解決したいことを、本人の言葉と実体験から引き出す。

やり方（**The Mom Test** の3原則）：
1. 自分（Claude）のアイデアを売り込まない。ユーザーの生活・実体験を聞く。「何を作りたい？」より「直近で〜したとき、何に困った？」。
2. 仮定でなく過去の具体を聞く。「〜があったら使う？」は誰でも Yes と言うので無意味。「最後に〜したのはいつ、何が起きた？」と聞く。
3. 話しすぎず聞く。

深掘りのコツ：
- 不満が出たら「なぜそれが嫌か」を繰り返し問う（5 Whys 的）。表層の不満の下に、機能的・社会的・感情的な動機（JTBD）が隠れている。
- 複数の不満が出たら、共通する根っこを探す。
- ✅ 出てきた不満・動機をユーザーの言葉のまま控える（後で journal とコンセプトに使う）。
- ❌ 1つ目の不満が出た時点で解決策の話に飛ばない。

### Phase 2 — コンセプトに収束（問題空間を定義 / Define）

目的：広げた不満を貫く1本の軸（コアコンセプト）に絞る。

やり方：
- 集めた不満を眺め、「これらは結局○○という1つの問題だ」という仮説を立ててユーザーにぶつける。
- ユーザーが「しっくりくる」と言う表現が見つかるまで言い換える。**コンセプトはユーザーの腹落ちが全て。**
- コンセプトは一文で言えるまで磨く（例：「本を厚みと位置を持つ物体として扱う」）。

問題が一文になると、以降すべての判断基準になる。

### Phase 3 — 解決策を広げて柱を決める（解決空間を発散→収束 / Develop→Deliver）

目的：コンセプトに沿った機能アイデアを広げ、最初に作る「柱」を1つ選ぶ。

やり方：
- コンセプトを軸に複数の機能アイデアを出す。
- **各アイデアがコンセプトとどう繋がるかを示す。** バラバラの機能リストにせず、1本の軸から全部が導かれる状態を目指す。
- ユーザーに「一番ワクワクするのは／違和感があるのは？」を聞き、MVP の柱を1つに絞る。
- 重要な設計トレードオフがあれば、この段階で明示して合意する。

### 区切りごとに journal に記録（R-001）

Phase 1〜3 の区切りや重要な意思決定の直後に `journal/YYYY-MM-DD-{topic}.md` へ残す。記録対象：出した不満・アイデアと背景／狙い、採用・不採用の理由、コンセプト、設計判断とその理由、次の宿題。**コードを読めば分かる実装詳細は書かない。**

### Phase 4 — コンセプトの裏付けを調査（R-003）

目的：コアコンセプトに、エビデンスのある知見で裏付けを与える。

やり方：
- コンセプトのキーワード（例：空間記憶、習慣化、認知負荷）で WebSearch し、論文・メタ分析・一次情報を当てる。
- 知見を「コンセプトの正しさ」と「個々の機能の根拠」に結びつける。
- 出典（論文名・著者・年・URL）を必ず控える。

なぜ：「なんとなく良さそう」を「研究がこう言っている」に変えると、企画の説得力と設計判断の質が上がる（R-003 の精神）。

### Phase 5 — 企画資料を生成

目的：ここまでを再利用可能な企画資料に落とす。テンプレートと書き方は [references/doc-templates.md](references/doc-templates.md) を参照。

生成物：
- `{app}/README.md` — 概要・コアコンセプト・差別化の核
- `{app}/docs/concept.md` — 差別化コンセプト + エビデンス（出典付き）
- `{app}/docs/features.md` — 機能一覧（フェーズで優先度を表現）
- `{app}/roadmap/README.md` — 設計哲学・（決まっていれば技術スタック）・フェーズ概要・MVP 完成基準
- 必要なら `{app}/roadmap/phaseN-*.md` — 実装ステップ・データモデル・完成基準

書き方の要点：
- 全機能を1本の軸（コンセプト）から導く。
- concept.md は「問題 → エビデンス → 知見と機能の対応表 → 出典」の順。
- roadmap は MVP を**現実的なスコープに絞る**（最初から全部入れない。実装難度の高い要素は後フェーズへ）。

## 進め方の注意

- 一度に全部やろうとせず、フェーズの区切りでユーザーに方向を確認する。発散フェーズでは自由回答で広げ、収束フェーズで選択肢を絞る。
- ユーザーが初心者の領域（言語・技術）なら、企画後の技術選定・実装計画も基礎から丁寧に。
- 繰り返し使えるサブ作業を見つけたら skill 化を提案する（R-002）。
