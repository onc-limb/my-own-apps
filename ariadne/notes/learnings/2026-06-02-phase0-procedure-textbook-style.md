# procedure を「教科書 + 写経」スタイルに改稿する方針

## コンテキスト
学習者から「procedure に例文を少し書くだけでなく、そもそもどういう原理でどう考えてそうなるのかを教科書的に記載し、写経的に学べるようにしてほしい。最後に理解度確認問題を用意して」という要望。
対象は procedure 全 34 ファイル、phase ごとに順番に改稿する方針で合意。phase0（01〜04）を完了。

## 確定したテンプレート（後続フェーズも踏襲する）

各ファイル冒頭:
- **Goal**: 「構文暗記でなく原理理解」を明記
- **このドキュメントの読み方**: 写経の作法（自分で打つ・型を読む・改造する のリズム）
- 必要なら「全体像/地図」を先に置く（例: ツールチェイン表、再帰の組み立て方）

各節（学習トピック）の構造 = **なぜ学ぶか → 原理（考え方・図解・例え・型の読み下し）→ 写経（utop 応答コメント付き）→ 観察ポイント → つまずき**

ファイル末尾:
- **まとめ表**: 学んだことが次フェーズでどう効くかを接続
- **Verification**: 自己チェック（手で・見ずにできるか）
- **理解度確認問題**: 解答は載せない（R-001）。「解いたらレビュー」方式と明記
- **Pitfalls / Tips**, **Outputs**, **Next**

## 重要な設計判断
- **解答は載せない**（学習者合意 + R-001）。確認問題は「自分で試し、答えと理由をメモしてからレビュー依頼」と毎回案内する。
- **写経用コード（01〜03）と演習（04）を区別する**:
  - 概念学習フェーズ（環境構築・文法・モジュール）は写経コードを充実させてよい（ユーザーが「写経的に」と明示指示）。
  - **演習フェーズ（04 Practice 等）は完成コードを載せず、型定義 + ベースケースのヒント + 考え方に絞る**（R-001/R-002。答えの先出しを避ける）。型定義は「設計の対象」なので示してよい。

## なぜこうするか
- 「コード + 一行説明」では原理が頭に残らない。動機（なぜ）→原理→手を動かす→観察、の順が定着しやすい。
- 型を読む習慣づけを最優先にした（OCaml 学習の本質）。
- 後フェーズ（AST 走査）への接続を各ファイルで明示し、学習の見通しを持たせる。

## 次にやること
- phase1〜5 を同じテンプレートで順次改稿。phase ごとに区切ってレビューを仰ぐ。
- phase1 以降は実装比重が上がるため、04 と同様「完成コードを骨組みに留める」判断が増える見込み。AST 走査ロジックそのものは学習者が書くべき対象なので、骨組み + 考え方に徹する。
