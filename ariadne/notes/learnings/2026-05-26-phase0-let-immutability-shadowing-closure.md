# `let` と不変性 — 束縛・シャドーイング・クロージャ

## コンテキスト
Phase 0 言語基礎の学習中、`let` の挙動を自分で実験して以下を確認した。

## 学習者の発見（重要）
1. `let x = 1` の `x` は不変（immutable）であり、後から値を変えられない
2. `let x = 2` と書き直すのは **「x を書き換える」のではなく「同じ名前で新しい束縛を作る」** だけ
3. 2回目の `let` の前に `x` を参照している関数を定義していた場合、
   その関数は **最初の `let` の値** を保持し続けることを実験で確認

→ この3点で、関数型言語の核心的な3概念に同時に触れた。

## 概念のラベル付け
| 学習者の観察 | 概念名 | 一言で |
|--------------|--------|--------|
| ① 変更できない | **イミュータブル束縛 (immutable binding)** | 名前と値は一度結びつくと変わらない |
| ② 同名で新しい束縛 | **シャドーイング (shadowing)** | 古い束縛は内側スコープで「隠される」だけで、まだ生きている |
| ③ 関数は最初の値を保持 | **レキシカルスコープ / クロージャ (closure)** | 関数は定義時に見えていた環境を閉じ込める |

## 思考モデルの更新

### Before（命令型言語のイメージ）
- 変数 = 中身を入れ替えられる **箱**
- `x = 1; x = 2` は「箱の中身を 1 から 2 に置換」

### After（OCaml のイメージ）
- 束縛 = 値に貼られた **名札**
- `let x = 1; let x = 2` は「1 に貼った名札 x を一度はがして、新しく 2 にも x の名札を貼る」
- 1 を指していた他の名札（関数の中で覚えている x）は影響を受けない

## クロージャの本質（学習者の観察 ③ の言い換え）
> 関数は **定義された時点** での束縛環境を覚えていて、**呼び出された時点** では覚え直さない

つまり関数オブジェクト = コード + その時点で見えていた環境（束縛の集合）。
これを **クロージャ (closure)** と呼ぶ。

## 残した宿題（次の実験ネタ）

### 実験 A: クロージャの確認（学習者自身が書く）
```text
let x = 10
let f = fun () -> x     (* 環境を閉じ込めた関数 *)
let x = 999             (* シャドーイング *)
f ()                    (* 何が返る？ *)
```
予想 → 実行 → 結果と理由を別ノートに書く。

### 実験 B: 本当に書き換えたいとき
OCaml には書き換え可能な箱が **別の仕組み** として存在する。
- キーワード: 参照セル (reference cell)
- 作成: `ref` プリミティブ
- 代入: `:=`
- 読み出し: `!`

→ Phase 0 のどこかで `ref` に遭遇する。そのとき本格的に学ぶ。

## なぜこの学びが効くか（先の見通し）
- Phase 1: AST 走査では「カウンタを再帰で持ち回す」「アキュムレータパターン」が頻出
- これは「変数を書き換える」のではなく「新しい束縛を作って次の再帰に渡す」スタイル
- 今日の理解がそのまま地続きで使われる
