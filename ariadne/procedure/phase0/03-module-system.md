# Phase 0 / 03 — Module System

## Goal
OCaml のモジュールとシグネチャの基本を理解し、後フェーズで「指標ロジックとパーサを分離する」設計の足場を作る。
ここでのゴールは構文暗記ではなく、**「実装を隠して、決めたインターフェースだけ外に見せる」という発想**を手に入れること。これが大きなコードを壊れにくく保つ鍵になる。

## Prerequisites
- [02 Language Fundamentals](02-language-fundamentals.md) 完了

## このドキュメントの読み方
モジュールは「コードを部屋に分ける」話。各節で **なぜ部屋に分けるのか → どう分けるか → 壁（インターフェース）をどう作るか** を追う。
写経しながら、特に「壁を作ると外から何が見えなくなるか」を**コンパイルエラーで体験する**こと。

---

## なぜモジュールが必要か（先に動機）

プログラムが大きくなると、必ず 3 つの問題が出る。

1. **名前の衝突**：あちこちで `make` や `get` を定義したい
2. **関心の混在**：パーサのコードと指標計算のコードがファイルにごちゃ混ぜになる
3. **内部実装への依存**：他人（未来の自分）が内部表現に直接触り、変更できなくなる

モジュールはこの 3 つを一度に解く道具：
- **名前空間**：`Counter.get` のように所属で区別 → (1) を解決
- **関心の分離**：機能ごとにファイル／モジュールを分ける → (2) を解決
- **カプセル化**：内部を隠し、公開インターフェースだけ見せる → (3) を解決

> Ariadne では最終的に「**パーサ（言語ごとに違う）**」と「**指標ロジック（言語共通）**」を分離したい。その分離をモジュールとシグネチャで実現する。今はその予行演習。

## Steps

### 1. インラインモジュール — `struct ... end`

モジュールは「型と関数をひとまとめにした箱」。最小形はファイル内に直接書ける。

```ocaml
module Counter = struct
  let make () = ref 0           (* 可変参照を作る。中身 0 *)
  let incr c = c := !c + 1      (* c := v は代入、!c は中身の読み出し *)
  let get c = !c
end

let () =
  let c = Counter.make () in
  Counter.incr c;
  Counter.incr c;
  Printf.printf "%d\n" (Counter.get c)   (* 2 *)
```

**原理**：`module Name = struct ... end` で箱を作り、中の値は `Name.値` でアクセスする。
（補足：ここでは `ref`＝可変参照を使っている。02 で「ref は例外的手段」と学んだ通り、本来は控えめに使う。ここはモジュール構文を見せるための例。）

**観察**：`Counter.make` のように、外からは必ず `Counter.` を付けて呼ぶ。これが名前空間。

### 2. ファイル分割によるモジュール — `foo.ml` ⇒ `Foo`

dune では **ファイル名がそのままモジュール名になる**（先頭大文字化）。`counter.ml` を置くだけで `Counter` モジュールができる。

```
playground/modules/
├── dune-project
└── bin/
    ├── dune
    ├── counter.ml   ← これが Counter モジュールになる（struct/end を書かなくてよい）
    └── main.ml      ← Counter.make で呼べる
```

- `counter.ml` には **`module Counter = struct` の「中身だけ」**を書く（`make`/`incr`/`get` の定義をそのまま）。`struct ... end` で囲む必要はない。**ファイルそのものが struct**。
- `main.ml` から `Counter.make ()` で呼べる。`open` や import は不要（同じプロジェクト内なら dune が繋ぐ）。

**原理**：これが OCaml の「1 ファイル＝1 モジュール」の規約。関心の分離が**ファイル分けだけで**実現する。

### 3. シグネチャ（`.mli`）— 壁を作って内部を隠す

ここが本節の核心。`counter.mli` を作ると、**外に見せるものを明示的に絞れる**。

```ocaml
(* counter.mli — 公開インターフェース（壁） *)
type t                    (* ★中身を書かない＝「抽象型」。外からは「t という型がある」としか分からない *)
val make : unit -> t
val incr : t -> unit
val get  : t -> int
```

**原理：抽象型 `type t`**
- `counter.ml` 側では `type t = int ref` のように実装を持つ。
- だが `.mli` で `type t`（中身なし）と書くと、**外部コードからは `t` が `int ref` だと見えなくなる**。
- 結果：外部は `Counter.make` を通してしか `t` を作れず、`!c` のような内部操作も封じられる。

これは 02 で出た「**不正な状態を表現不能にする**」の延長。**内部表現を隠すことで、不正な触り方をコンパイル時に禁止**できる。

**観察（必ず試す）**：`main.ml` で `Counter.get` の代わりに `!c`（中身の直接読み出し）を試みると、**コンパイルエラー**になる。「`t` の中身が見えないから直接触れない」ことを体で確認する。これが壁の効果。

> dune は `.mli` を自動で拾う。`bin/dune` の変更は不要。

### 4. ファンクタ — 「モジュールを引数に取るモジュール」

最後に軽く触る。ファンクタは**モジュールを受け取って、新しいモジュールを返す「モジュールの関数」**。

**なぜ要るか**：「中身の型は違うが、振る舞いは共通」なものを量産したいとき。
例：「int の集合」「string の集合」を別々に手書きするのは無駄。「**比較できる型なら何でも**集合を作れる仕組み」を一度書きたい。

```ocaml
(* 「比較できる」という契約（シグネチャ） *)
module type ORDERED = sig
  type t
  val compare : t -> t -> int
end

(* ORDERED を満たすモジュール O を受け取り、O の集合モジュールを作る *)
module MakeSet (O : ORDERED) = struct
  type elt = O.t
  type t = elt list
  let empty = []
  let add x s = if List.exists (fun y -> O.compare x y = 0) s then s else x :: s
end

(* int 用の集合を「生成」する。括弧の中が「比較できる契約」を満たす int モジュール *)
module IntSet = MakeSet (struct type t = int let compare = compare end)
```

**原理**：
- `module type ORDERED = sig ... end` は**モジュールの型（契約）**。「`t` 型と `compare` 関数を持て」と要求する。
- `MakeSet (O : ORDERED)` は「`ORDERED` を満たすモジュール `O` を入れると、それ用の集合モジュールを返す」関数。
- `MakeSet(...)` を呼ぶと、入れた型に合わせた新しいモジュール（`IntSet`）が**生成される**。

今は **「シグネチャ（契約）を満たすモジュールを差し込むと、それに合わせたモジュールができる」** と理解できれば十分。
Phase 2 以降で「**Parser モジュールを差し替えても指標ロジックは共通**」という設計でもう一度出会う。今は深追いしない。

### Q1. ファイルとモジュールの対応
`stack.ml` というファイルを作ると、外からは何という名前で呼べるか。`struct ... end` を書く必要はあるか。

### Q2. 抽象型の効果（重要）
`counter.mli` の `type t` を、もし `type t = int ref` と**中身まで書いた**ら、外部コードからは何ができるようになってしまうか。「壁が無くなる」とはどういうことか、具体的に述べよ。

### Q3. 隠蔽の確認
`counter.mli` から `val incr : t -> unit` の行を**削除**したら、`main.ml` の `Counter.incr c` はどうなるか予想し、実際に試して確かめよ。なぜそうなるか説明せよ。

### Q4. シグネチャ＝契約
`module type ORDERED` は「契約」だと説明した。`compare` を持たないモジュールを `MakeSet` に渡したら何が起きるか予想せよ。これは 02 で学んだ「網羅性検査」と似た安全性だが、何を保証しているか。

### Q5. 設計への接続
Ariadne で「パーサ（言語ごとに違う）」と「指標ロジック（共通）」を分離したい。これを **抽象型・シグネチャ・ファンクタのどれで**実現できそうか、現時点の理解で仮説を立てよ（正解でなくてよい。考えることが目的）。

## Pitfalls / Tips
- `.mli` を書くと、`.ml` 側にあっても **`.mli` に書いていない値は外から使えなくなる**（隠蔽が効く）。「なぜか関数が見つからない」は `.mli` の書き忘れが多い。
- dune は `.mli` を自動で拾うので、`dune` ファイルへの明示記述は不要。
- ファンクタは**今は深追いしない**。Phase 2 で「Parser を差し替える」場面で実感を持って再会する。
- `module type`（モジュールの型）と `module`（モジュール本体）を混同しない。前者は契約、後者は実体。
