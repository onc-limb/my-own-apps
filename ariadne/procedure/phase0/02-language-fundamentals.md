# Phase 0 / 02 — Language Fundamentals

## Goal
OCaml の文法と「発想」（不変・代数的データ型・パターンマッチ・Option/Result）を、後フェーズの AST 走査に転用できるレベルで身につける。
ここでの目標は「構文を覚える」ことではなく、**OCaml がなぜそういう設計になっているのか**を理解し、自分の手で再現できるようにすることである。

## Prerequisites
- [01 Environment Setup](01-environment-setup.md) 完了
- `utop` が起動できる

## このドキュメントの読み方（写経の作法）

このドキュメントは「読む」だけでは身につかない。次のリズムで進めること。

1. **原理を読む** — なぜそうなるのか、どう考えるのかを先に頭に入れる
2. **写経する** — コードを `utop` に**自分で打ち込む**（コピペしない。打つことで構文が手に馴染む）
3. **観察する** — utop が返す**型と値**を必ず読む。「期待と違う」が一番の学び
4. **改造する** — 1 箇所変えて、出力がどう変わるかを予想 → 確認する

> utop は式を評価すると `- : 型 = 値` の形で返す。**型を読む習慣**を最優先で身につけること。OCaml の学習はほぼ「型を読めるようになる」ことに等しい。

各節は **なぜ → 原理 → 写経 → 観察ポイント → つまずき** の順で構成している。

## 1. `let` と不変性 — 「変数」ではなく「束縛」

### なぜこれを学ぶか
OCaml には、多くの言語にある「変数への再代入（`x = x + 1`）」が**標準では存在しない**。
この一点が、後で AST を安全に走査するコードの書き方を根本から決める。まずここで世界観を切り替える。

### 原理
`let x = 1` は「箱 `x` に 1 を入れる」のではなく、「**名前 `x` を値 1 に結びつける（束縛する／bind）**」という意味。
一度結びついた `x` は、その後ずっと 1 のまま変わらない。これを **不変（immutable）** という。

ではこれは何に見えるか。

```ocaml
let x = 1;;
let x = x + 1;;
```

2 行目は「`x` を書き換えた」のではない。「**新しい `x`**（値 2）を、たまたま同じ名前で**新しく束縛した**」だけ。
古い `x`（値 1）はまだ存在していて、新しい `x` が前の `x` を**覆い隠している**。これを **シャドーイング（shadowing）** と呼ぶ。

> 例えるなら、再代入は「同じホワイトボードを消して書き直す」、シャドーイングは「新しいホワイトボードを古いものの手前に立てかける」。古い板は消えていない。

### 写経
```ocaml
let x = 1;;
(* utop の応答: val x : int = 1  ← 型 int、値 1 *)

let y = x + 10;;
(* val y : int = 11  ← この時点の x（=1）を使って計算 *)

let x = x + 1;;
(* val x : int = 2  ← 新しい x を束縛。古い x は隠れる *)

y;;
(* - : int = 11  ← y は古い x で計算済みなので 11 のまま。影響を受けない *)
```

### 観察ポイント
- `y` が **11 のまま**であることを必ず確認する。もし再代入だったら `y` も道連れに変わるはず。変わらない＝**過去の計算結果は壊れない**、という不変性の恩恵。
- これは後フェーズで効いてくる。AST を走査するとき「途中で値が書き換わっていないか」を疑わなくてよい＝**バグの温床がひとつ消える**。

### つまずき
- 「シャドーイングがあるなら結局再代入と同じでは？」→ 違う。シャドーイングは**新しい名前の束縛**であり、既に作られた値（上の `y`）には一切影響しない。再代入は値そのものを破壊する。
- 本当に書き換えたい場合は `ref`（可変参照）を使うが、OCaml では**例外的手段**。基本は使わない。

## 2. 関数定義 — 関数は「1 引数の関数の連なり」

### なぜこれを学ぶか
OCaml の関数は、見た目は多引数でも内部では**1 引数関数の連鎖**として扱われる（カリー化）。
これを理解すると「部分適用」という強力な道具が使え、後で走査関数を組み立てるときに効く。

### 原理
```ocaml
let add x y = x + y;;
```
これは「2 引数の関数」に見えるが、OCaml は内部的にこう解釈する：

> `add` は「`x` を受け取ると、**『`y` を受け取って `x + y` を返す関数』** を返す関数」

型を見ると一目瞭然：

```
val add : int -> int -> int
```

`->` は右結合なので、これは `int -> (int -> int)` と読む。「int を渡すと（int を渡すと int を返す関数）が返る」。
これを **カリー化（currying）** という。

この設計の利点が **部分適用（partial application）**：引数を全部渡さず、途中まで渡して「新しい関数」を作れる。

```ocaml
let add5 = add 5;;
(* val add5 : int -> int  ← add に 5 だけ渡した。残り 1 引数の関数になった *)
add5 10;;
(* - : int = 15 *)
```

### 写経
```ocaml
let add x y = x + y;;
(* val add : int -> int -> int *)

(* 名前付き引数: 呼ぶ側が引数の意味を明示できる。順番を入れ替えても良い *)
let sub ~from ~amount = from - amount;;
sub ~amount:3 ~from:10;;
(* - : int = 7  ← 順番を逆に書いても ~from が from に入る *)

(* オプション引数: 省略時のデフォルトを持つ。末尾に () が要る理由は後述 *)
let greet ?(name = "world") () = Printf.printf "Hello, %s\n" name;;
greet ();;            (* Hello, world *)
greet ~name:"Ada" ();; (* Hello, Ada *)

(* 部分適用 *)
let add5 = add 5;;
add5 100;;            (* - : int = 105 *)
```

### 観察ポイント
- `add` の型 `int -> int -> int` を声に出して「int を受け取ると、int を受け取って int を返す関数が返る」と読む。この読み方が体に入ると OCaml が一気に楽になる。
- `add5` の型が `int -> int` に**減っている**ことを確認。引数を 1 つ「消費」した分、型から `int ->` が 1 つ消えた。

### つまずき
- **オプション引数の後ろの `()` は何？** → オプション引数は「省略できる」ため、`greet` とだけ書くと「引数待ちの関数」なのか「引数省略で呼んだ」のか OCaml が判断できない。末尾に省略不可能な引数（ここでは `unit` 型の `()`）を置くことで「ここで呼び出し確定」と伝える。`greet ()` の `()` がそれ。
- 名前付き引数 `~from` は「ラベル `from` を持つ引数」。`sub ~from:10` で「ラベル from に 10 を渡す」。ラベル名と変数名が同じなら `sub ~from` と短縮できる。

## 3. リスト処理 — 再帰と高階関数

### なぜこれを学ぶか
AST は「子ノードのリスト」を大量に持つ構造。**リストを再帰でたどる**のが Phase 1 以降の中心作業になる。
ここで「ループではなく再帰で考える」発想と、`map`/`filter`/`fold` という 3 つの基本道具を手に入れる。

### 原理：リストは再帰的なデータ
OCaml のリストは、見た目は `[1; 2; 3]` だが、本当の正体は**入れ子の構造**：

```
1 :: (2 :: (3 :: []))
```

- `[]` は **空リスト**（終端）
- `x :: xs` は「**先頭 `x`** と **残り `xs`**（これもリスト）」をつなぐ演算子（`::` は cons と読む）

つまりリストは「空」か「先頭＋残りのリスト」のどちらか。**この 2 択構造がそのまま再帰の形になる**。

```ocaml
let rec sum = function
  | [] -> 0                 (* 空なら合計は 0（ベースケース） *)
  | x :: xs -> x + sum xs   (* 先頭 + 「残りの合計」（再帰ケース） *)
```

`sum [1;2;3]` の展開を頭の中で追う：
```
sum [1;2;3]
= 1 + sum [2;3]
= 1 + (2 + sum [3])
= 1 + (2 + (3 + sum []))
= 1 + (2 + (3 + 0))
= 6
```
**「残りの問題を自分自身で解く」**——これが再帰の発想。`rec` は「この関数は自分を呼ぶ」という宣言。

### 原理：高階関数（関数を受け取る関数）
上のような再帰は頻出パターンなので、標準ライブラリが部品化している。それが `map` / `filter` / `fold`。
共通点は「**関数を引数に取る**」こと（＝高階関数）。

| 道具 | やること | イメージ |
|------|----------|----------|
| `List.map f xs` | 各要素に `f` を適用して**新しいリスト**を作る | 各要素を変換 |
| `List.filter p xs` | 述語 `p` が `true` の要素**だけ**残す | ふるい分け |
| `List.fold_left f init xs` | 左から順に**1 つの値に畳み込む** | 集計・合算 |

### 写経
```ocaml
List.map (fun x -> x * 2) [1; 2; 3];;
(* - : int list = [2; 4; 6]  ← 各要素を 2 倍 *)

List.filter (fun x -> x mod 2 = 0) [1; 2; 3; 4];;
(* - : int list = [2; 4]  ← 偶数だけ残る *)

List.fold_left (fun acc x -> acc + x) 0 [1; 2; 3; 4];;
(* - : int = 10  ← acc(累積) に順に足し込む。初期値 0 *)

(* fold は最強。sum も map も filter も fold で書ける。
   fold_left の f は (累積値 acc, 次の要素 x) を受け取り、新しい累積値を返す *)
```

`fold_left` の動きを展開して追う：
```
fold_left (+) 0 [1;2;3]
→ acc=0, x=1 → 1
→ acc=1, x=2 → 3
→ acc=3, x=3 → 6
```

### 観察ポイント
- `map` の結果の型が `int list` であること、**元のリストは変わっていない**こと（不変！map は新しいリストを返す）。
- `fold_left` の第 1 引数 `f` は **2 引数**（`acc` と `x`）。これが「畳み込み」の核心。`acc` に結果を溜めていく。

### つまずき
- **`=` と `==` の違い**：OCaml で値の比較は `=`（構造的等価）。`==` は「同じメモリか」の物理等価で、普段は使わない。`x mod 2 = 0` の `=` は比較演算子。
- `let rec` の `rec` を忘れると「`sum` が未定義」と怒られる。再帰関数には必ず `rec`。

## 4. タプルとレコード — 「複数の値をまとめる」2 つの型

### なぜこれを学ぶか
「位置とサイズ」「行番号と内容」のように、複数の値を 1 セットで扱いたい場面は多い。
OCaml には目的別に 2 つの「まとめ方」がある。使い分けの感覚を掴む。

### 原理：これらは「直積型（product type）」
タプルもレコードも「**A かつ B かつ C**」を 1 つにまとめた型。数学の直積に対応するので**直積型**と呼ぶ。

- **タプル** `(1, "a")`：順番で区別。軽量・無名。型は `int * string`。
- **レコード** `{ x = 1; y = 2 }`：**名前（フィールド）**で区別。可読性が高い。要事前型定義。

> 使い分けの目安：要素が 2〜3 個で意味が自明（座標など）→ タプル。意味に名前を付けたい・要素が増える → レコード。

### 写経
```ocaml
(* タプル: 位置で区別 *)
let pair = (1, "a");;
(* val pair : int * string = (1, "a") *)

let (n, s) = pair;;   (* 分解（destructuring）。n=1, s="a" *)

(* レコード: 名前で区別。先に型を定義する *)
type point = { x : int; y : int };;

let p = { x = 1; y = 2 };;
(* val p : point = {x = 1; y = 2} *)

p.x;;                 (* - : int = 1  ← ドットでフィールドアクセス *)

let { x; _ } = p;;    (* レコードも分解できる。_ は「残りは無視」 *)
(* val x : int = 1 *)
```

### 観察ポイント
- タプルの型 `int * string` の `*` は「直積」の記号。`int` と `string` の組、という意味。
- `let { x; _ } = p` の `_` を消すと「`y` を束縛していない」警告が出る。`_` は「あえて捨てる」明示。

### つまずき
- レコードのフィールドアクセスは `p.x`、タプルは位置分解（`let (n, _) = pair`）。タプルに `.0` のような添字アクセスは**ない**。
- 同名フィールドを持つ型が複数あると、OCaml はどの型か推論に迷うことがある。その場合 `(p : point).x` のように型注釈で助ける。

## 5. 代数的データ型（ADT）と網羅パターンマッチ — Phase 1 以降の心臓部

### なぜこれを学ぶか
**ここが Phase 0 で一番重要**。AST（抽象構文木）はまさに ADT で表現される。
「式は、整数か、足し算か、掛け算か……のどれか」という**「いずれか（or）」の構造**を型で表し、それを**漏れなく場合分け**するのがパターンマッチ。この 2 つで AST 走査の 8 割が書ける。

### 原理：ADT は「直和型（sum type）」
4 節の直積型が「A **かつ** B」なら、ADT は「A **または** B または C」を表す。これを**直和型**と呼ぶ。

```ocaml
type expr =
  | Int of int            (* 整数リテラル。int を 1 つ持つ *)
  | Add of expr * expr    (* 足し算。左辺と右辺（どちらも expr！）を持つ *)
  | Mul of expr * expr    (* 掛け算。同上 *)
```

ポイント 2 つ：
1. `Int` / `Add` / `Mul` を **コンストラクタ（構築子）** と呼ぶ。`expr` 型の値は**この 3 つのいずれか**。
2. `Add of expr * expr` のように、定義の中に `expr` 自身が出てくる＝**再帰的な型**。だから木構造を表せる。`Add (Int 1, Mul (Int 2, Int 3))` のように入れ子にできる。

### 原理：パターンマッチと網羅性検査
ADT を処理するときは `match`（または `function`）で**コンストラクタごとに場合分け**する。

```ocaml
let rec eval = function
  | Int n -> n
  | Add (a, b) -> eval a + eval b   (* 左右を再帰的に評価して足す *)
  | Mul (a, b) -> eval a * eval b
```

`Add (a, b) ->` は「もし値が `Add` なら、中身を `a` と `b` に取り出して」という**分解と束縛を同時に行う**記法。

そして OCaml の最大の武器：**コンパイラが「全ケースを尽くしているか」を検査する**（網羅性検査）。
`Mul` の行を消すと：

```
Warning 8: this pattern-matching is not exhaustive.
Here is an example of a case that is not matched: Mul (_, _)
```

「`Mul` の場合を書き忘れてるよ」とコンパイラが教えてくれる。
**これは「型に新しいケースを足したとき、対応漏れを全部コンパイラが指摘してくれる」ことを意味する**——大規模な AST 処理で命綱になる安全装置。

### 写経
```ocaml
type expr =
  | Int of int
  | Add of expr * expr
  | Mul of expr * expr;;

(* 1 + (2 * 3) を表す木を組み立てる *)
let e = Add (Int 1, Mul (Int 2, Int 3));;

let rec eval = function
  | Int n -> n
  | Add (a, b) -> eval a + eval b
  | Mul (a, b) -> eval a * eval b;;

eval e;;
(* - : int = 7 *)
```

### 観察ポイント
- `eval e` を手で展開：`Add(Int 1, Mul(Int 2, Int 3))` → `eval(Int 1) + eval(Mul(...))` → `1 + (2*3)` → `7`。**型の再帰構造と関数の再帰が対応している**ことを体感する。
- **意図的に網羅性警告を起こす**：`Mul` の行をコメントアウトして再評価し、Warning 8 を自分の目で見る。これが後で何度も助けてくれる。

### つまずき
- **ワイルドカード `_` の罠**：`| _ -> 0` のように書くと全ケースを吸収するので警告が消える。**だが新しいコンストラクタを追加しても気付けなくなる**。AST 処理では「`_` で握りつぶさず、コンストラクタを明示的に列挙する」のが鉄則。`_` は本当に「どうでもいい残り」のときだけ。
- `match` の各行頭の `|` 揃え忘れ、最初の行の `|` 有無で構文エラーが出やすい。

## 6. Option / Result — 「失敗」を型で表す

### なぜこれを学ぶか
「ファイルが無い」「リストに見つからない」「0 除算」——失敗は必ず起きる。
他言語の `null` / `nil` は「失敗を型に書かず黙って忍ばせる」ため、参照漏れがバグになる（いわゆる「10 億ドルの過ち」）。
OCaml は **失敗の可能性を型に明記**し、コンパイラに「対応した？」と確認させる。

### 原理：Option（あるか / ないか）
```ocaml
type 'a option = None | Some of 'a   (* これは標準で定義済み。自分で書く必要はない *)
```
これも ADT（5 節）。`'a`（任意の型）の値が「ある（`Some x`）」か「ない（`None`）」かを表す。
`'a` は**型変数**（ジェネリクス）。`int option` なら `Some 3` か `None`。

「見つかったかどうか分からない検索」は `option` を返す：
```ocaml
List.find_opt (fun x -> x > 2) [1; 2; 3];;
(* - : int option = Some 3 *)
List.find_opt (fun x -> x > 9) [1; 2; 3];;
(* - : int option = None *)
```
受け取った側は `match` で「ある場合／ない場合」の両方を書かされる。**書き忘れは網羅性警告で弾かれる**＝null 参照漏れが構造的に防げる。

### 原理：Result（成功 / 失敗の理由つき）
`option` の `None` は「なぜ無いか」を語らない。**失敗の理由を持ちたい**ときは `Result`：
```ocaml
type ('a, 'e) result = Ok of 'a | Error of 'e   (* これも標準定義済み *)
```
成功なら `Ok 値`、失敗なら `Error 理由`。

### 写経
```ocaml
let safe_div a b =
  if b = 0 then Error "divide by zero"
  else Ok (a / b);;
(* val safe_div : int -> int -> (int, string) result *)

match safe_div 10 2 with
| Ok v -> Printf.printf "ok: %d\n" v
| Error msg -> Printf.printf "err: %s\n" msg;;
(* ok: 5 *)

match safe_div 10 0 with
| Ok v -> Printf.printf "ok: %d\n" v
| Error msg -> Printf.printf "err: %s\n" msg;;
(* err: divide by zero *)
```

### 観察ポイント
- `safe_div` の戻り型が自動で `(int, string) result` と推論されること。**「失敗しうる」が型に現れている**のがポイント。呼ぶ側はこの型を見ただけで「失敗処理が要る」と分かる。
- `match` から `Error` の行を消すと網羅性警告。**失敗処理の書き忘れがコンパイル時に露見する**。

### つまずき
- いつ Option でいつ Result か → 「なぜ失敗したかを呼び出し側に伝えたい」なら Result、「単に有無だけ」なら Option。
- `Option.value ~default:0 (Some 3)`、`Result.map` など便利関数がある。**型から使い方を推測する練習**をすると、ドキュメント無しで使えるようになる（Verification 参照）。

## 7. 例外と Result の使い分け — 設計判断

### なぜこれを学ぶか
OCaml にも例外 (`raise`, `try ... with`) はある。だが「何でも例外」にすると、失敗が型に現れず 6 節の利点が消える。**いつ型（Result）でいつ例外か**の方針を自分の言葉で持つ。

### 原理：境界線
| 状況 | 手段 | 理由 |
|------|------|------|
| 想定内の失敗（ファイル無し・パース失敗・入力不正） | `Result` | 呼び出し側が**回復・対処すべき**もの。型に出して対応を強制 |
| 真にバグ・復帰不能（不変条件違反・内部状態の破壊・「ここには絶対来ない」） | 例外 | 対処しようがない。早く落として原因を晒すべき |

> 合言葉：「**呼び出し側が対処できる失敗は値（Result）で返す。対処しようのない異常は例外で落とす**」。

### 写経（方針を試す）
```ocaml
(* 想定内 → Result *)
let parse_int s =
  match int_of_string_opt s with
  | Some n -> Ok n
  | None -> Error (Printf.sprintf "not an int: %s" s);;

parse_int "42";;    (* - : (int, string) result = Ok 42 *)
parse_int "abc";;   (* Error "not an int: abc" *)

(* 復帰不能 → 例外（assert は不変条件チェックの例） *)
let head_exn = function
  | [] -> failwith "head_exn: empty list (caller's bug)"
  | x :: _ -> x;;
```

### つまずき / Tips
- `int_of_string "abc"` は**例外**を投げるが、`int_of_string_opt "abc"` は `None` を返す。標準ライブラリには `_opt` 版（Option を返す安全版）と例外版が両方ある。学習中は `_opt` 版を優先する。
- 自分なりの一行ルールを `cheatsheet.md`（後述の Outputs）にメモすること。「想定内＝Result／バグ＝例外」を自分の言葉で書き直す。

## Pitfalls / Tips（全体）
- セミコロン `;`（式の区切り）と二重セミコロン `;;`（**対話環境で「ここまで評価」の合図**）を混同しない。`.ml` ファイルでは基本 `;;` は不要。
- `match` の `|` の付け忘れ・インデント崩れで型エラーが出やすい。エラーは**行番号と「期待した型／実際の型」**を読む。
- `Printf.printf "%s\n"` のフォーマット指定子は型と一致が必要（`%d`=int, `%s`=string）。**コンパイル時に**チェックされる。
