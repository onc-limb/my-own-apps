# daedalus — コンセプト

## 解きたい課題

「こういうインフラが欲しい」という意図から、実際に動く Terraform 構成を作って
デプロイするまでには、HCL の記述・プロバイダ仕様の確認・`plan`/`apply` のエラー潰し
という反復作業がある。とくに **エラーを読んで直して再実行** のループは退屈で時間を食う。

daedalus は、この「生成 → デプロイ → エラー修正 → 再デプロイ」のループを自律化する。

## 中核アイデア：ループを作らない

エージェント開発でいちばん作り込みがちなのが「LLM にツールを使わせるループ」だが、
それは **ヘッドレス Claude Code がすでに持っている**。Claude Code は Bash で `terraform` を
実行し、stderr を読み、`.tf` を編集し、再実行する——というツール使用ループを自律的に回せる。

そこで daedalus は **Claude Agent SDK でヘッドレス Claude Code を制御**する設計を採る。
自前で実装するのは次の3点だけに絞る:

1. **構成入力** — spec(YAML) → タスクプロンプト変換
2. **ガードレール** — `apply`/`destroy` のゲートと危険シェルの遮断
3. **記録** — 試行履歴（イベント・エラー・コスト）の保存

> この「ヘッドレス Claude Code の上に薄いガードレールと記録を載せる」型は、
> 他のエージェントにも転用が効く。2 回以上使うようなら skill 化を検討する（ルート R-002）。

## なぜ Python なのか

`claude -p`（ヘッドレス）を使う構成では、Anthropic **API SDK** の言語差は意味を失う。
一方、ヘッドレス Claude Code をプログラムから制御する **Claude Agent SDK は Python / TS のみ**。
hook による権限制御・記録を素直に書けるため Python を採用した。

（検討の全経緯はルートの `journal/2026-06-10-cloud-infra-agent-decision.md` に記録。）

## スコープ（v1）

- 対象は単一スタック。spec は自然言語を許容する緩い YAML。
- **デフォルトは dry-run（plan のみ）**。`--apply` で実適用、`--allow-destroy` で破棄を解禁。
- state はローカルバックエンド前提（検証用）。

## 非スコープ（v1 では扱わない）

- マルチ環境（dev/stg/prod）の出し分け、リモート state、CI 連携
- コスト見積もりの自動ゲート（将来 PreToolUse フックで `plan` のコストを評価する余地あり）
- 複数スタックの依存管理
