# daedalus — 機能アイデア（5つの目線）

> daedalus（クラウドインフラ構築エージェント）に「あったら便利／用意した方が良さそう」な機能を、
> **ユーザー / 作者 / 管理者 / エンジニア / ビジネス** の5目線でブレストしたもの。
> 各項目は現状アーキテクチャ（Terraform エージェント・plan/approval/auto モード・PreToolUse/PostToolUse
> フック・GitHub 連携・FastAPI GUI・マルチクラウド）に紐づけ、**実装の足がかり**まで添えている。
>
> これは確定ロードマップではなく**アイデアの蓄積**。実装するものは別途取捨選択する（ルート R-002 の
> skill 化判断と同じく、繰り返し効くものから優先）。

凡例: 💡=アイデア / 🪝=既存の拡張ポイントにそのまま乗る / 🧱=新規モジュールが要る

---

## 1. ユーザー目線（インフラを作ってほしい人）

「spec を書く」「plan を理解する」「安心して承認する」のハードルを下げる機能。

| 機能 | 概要 | なぜ嬉しい | 実装の足がかり |
|---|---|---|---|
| **spec テンプレートギャラリー** 🪝 | よくある構成（静的サイト / VPC / k8s / DNS）の spec 雛形を選んで埋めるだけ | 白紙からの spec 記述が要らない | `examples/` を増やし、GUI にテンプレ選択を追加 |
| **対話型 spec ビルダー** 🧱 | 「何を作りたい?」を数問の質問で聞き、spec YAML を生成 | YAML の文法を知らなくても始められる | GUI に簡易ウィザード。生成は Claude に投げる |
| **plan の人間向けサマリ** 🪝 | `terraform plan` の追加/変更/削除を平易な日本語に要約して承認カードに表示 | 「何が起きるか」が一目で分かる | PostToolUse で plan 出力を整形（既に tail は保持済み） |
| **コスト概算の事前表示** 🧱 | apply 前に月額概算を提示（[Infracost](https://www.infracost.io/) 連携） | 「いくらかかる?」が承認前に分かる | PreToolUse で `terraform plan -out` → infracost 実行 |
| **アーキテクチャ図の自動生成** 🧱 | 作る構成を図（Mermaid / diagram）で可視化 | 文章より直感的にレビューできる | plan/`.tf` から Claude に図を生成させ GUI 表示 |
| **完了・承認依頼の通知** 🪝 | Slack / メール / Discord に「承認して」「完了した」を飛ばす | 画面に張り付かなくて良い | journal listener に通知シンクを追加 |
| **既存インフラのインポート** 💡 | 手動で作ったリソースを `terraform import` でコード化する支援 | 既存環境からでも始められる | spec に `import:` を足し、プロンプトで誘導 |
| **ワンクリック・ロールバック** 💡 | 直前の成功 state / コミットへ戻す | 失敗時に安心 | GitHub の履歴 + state スナップショットから復元 |

## 2. 作者目線（daedalus を育てる開発者）

エージェントの品質を計測し、改善ループを回すための機能。

| 機能 | 概要 | なぜ嬉しい | 実装の足がかり |
|---|---|---|---|
| **run メトリクス集計** 🪝 | 成功率・平均ループ回数・コスト・所要時間・却下率を集計 | どこが弱いか定量的に分かる | `.daedalus/runs/*.jsonl` を集計するコマンド |
| **system prompt のバージョン管理** 🪝 | プロンプト改訂を記録し、どの版でどの成績かを追える | 改善が前進か後退か判別できる | `prompts.py` に版番号 + run に記録 |
| **run のリプレイ / 回帰テスト** 🧱 | モック terraform で「ゴールデン spec」をE2E実行し、退行を検出 | プロンプト変更の安全網 | 既存のサーバE2Eテストを土台に拡張 |
| **失敗事例コーパス** 🪝 | エラー→修正の往復を収集し、頻出パターンを抽出 | プロンプト改善・skill 化の材料 | journal の `bash_post`/`Error:` を蓄積・分類 |
| **評価ハーネス（eval）** 🧱 | 代表 spec 群に対し apply 成功率・コストを定点観測 | モデル/プロンプト更新の影響を測れる | CI で nightly 実行（要 sandbox） |
| **プロバイダ拡張ポイント** 💡 | provider 別のヒント/ガードを差し込めるプラグイン構造 | 対応クラウドを増やしやすい | `guardrails.py` / `prompts.py` を provider 別に分岐 |
| **skill 昇華** 🪝 | 繰り返し出るレビュー観点を skill 化（ルート R-002/R-003） | 知見が再利用可能になる | journal の指摘ログから抽出 |

## 3. 管理者目線（チームで運用する人 / プラットフォーム管理者）

「誰が・何を・どこまで」を統制し、事故を防ぐためのガバナンス機能。

| 機能 | 概要 | なぜ嬉しい | 実装の足がかり |
|---|---|---|---|
| **GUI の認証・RBAC** 🧱 | ログイン必須化、ロール別に「実行可」「承認可」を分離 | 無認証 localhost 前提から脱却 | `server.py` に認証ミドルウェア + ロール |
| **監査ログ（改ざん耐性）** 🪝 | 誰が・いつ・どの run を承認/却下/apply したかを追記専用で記録 | コンプライアンス対応・事故調査 | journal を中央集約 + 署名/追記専用ストレージ |
| **ポリシーガードレール** 🪝 | 禁止リソース・必須タグ・許可リージョン・命名規則を強制（[OPA/Conftest](https://www.openpolicyagent.org/) や Sentinel） | 組織ルールを機械的に担保 | PreToolUse で plan に対しポリシー評価→deny |
| **予算ガード** 🪝 | 概算が閾値超過なら自動で承認必須化 or block | 想定外の高額リソースを止める | コスト概算（§1）+ `classify_bash` を拡張 |
| **シークレット管理連携** 🧱 | Vault / SOPS / クラウドのSecret Manager から env を安全注入 | キーの平文取り回しを避ける | 実行前に secrets backend から env を解決 |
| **リモート state + ロック** 🧱 | state を S3/GCS/TFC に置き、同時実行をロック | 複数人運用で state 破損を防ぐ | spec の `backend` を local 以外に対応 |
| **多段・ロール別承認** 🪝 | 本番は「2人承認」「特定ロールのみ」等のワークフロー | 重要変更の統制 | 承認 future を複数承認待ちに拡張 |
| **同時実行キュー / 並列** 🧱 | 現状「同時1run」をキュー化 or プロジェクト単位で並列 | チーム規模で詰まらない | `RunManager` をキュー対応に |

## 4. エンジニア目線（DevOps / SRE がワークフローに組み込む）

CI/CD・GitOps・テスト・可観測性に効く機能。

| 機能 | 概要 | なぜ嬉しい | 実装の足がかり |
|---|---|---|---|
| **PR ベース GitOps** 🪝 | spec を PR → daedalus が plan を PR にコメント → merge で apply | レビュー文化に自然に乗る | 既存 `github_sync` + GitHub Actions / webhook |
| **CI/CD 統合（ヘッドレス）** 🪝 | `daedalus run --mode plan` を CI で回し、差分を出力 | 自動チェックに組み込める | 既に headless 実行可。出力を機械可読に |
| **ドリフト検知** 🧱 | 定期 `plan` で実環境とコードの差分を検知し通知 | 手動変更の発見・是正 | スケジュール実行 + plan 差分の判定 |
| **セキュリティ/品質スキャン** 🪝 | apply 前に `tflint` / `tfsec` / `checkov` を必須実行 | 設定ミス・脆弱性を事前検出 | システムプロンプトに必須ステップ追加 + ガードで強制 |
| **構造化 plan の活用** 🪝 | `terraform show -json` で plan を型付きに解析 | 正規表現頼みの判定を堅くする | PostToolUse で `-json` を解析（現状は文字列マッチ） |
| **複数環境（dev/stg/prod）** 🧱 | 環境ごとの変数ファイル・workspace・承認ポリシー | 環境差分を安全に扱う | spec に `environments:` を追加 |
| **モジュール再利用** 💡 | 社内 Terraform モジュールレジストリを参照させる | 車輪の再発明を防ぐ | プロンプトにモジュール方針を注入 |
| **OpenTelemetry トレース** 🧱 | run をトレース化し、ログ/メトリクスを基盤に送る | 既存の可観測性基盤に統合 | journal を OTel エクスポータに接続 |
| **ローカル再現性** 🪝 | 同じ workspace / state を引き継いで再実行 | デバッグしやすい | workspace を run キーで永続化 |

## 5. ビジネス目線（意思決定者 / マネージャー）

コスト・リスク・ガバナンス・価値を経営判断につなげる機能。

| 機能 | 概要 | なぜ嬉しい | 実装の足がかり |
|---|---|---|---|
| **コストレポート / 予測** 🧱 | プロジェクト別・月次のコスト実績と予測をダッシュボード化 | 予算管理・投資判断の材料 | Infracost + クラウド課金API を集計 |
| **工数削減の定量化** 🪝 | 「手作業 vs daedalus」の所要時間・試行回数を可視化 | ROI を説明できる | run メトリクス（§2）から算出 |
| **ガバナンス / 監査証跡** 🪝 | 誰が承認したか・ポリシー遵守状況のレポート | SOC2/ISO 等の監査対応 | 監査ログ（§3）をレポート化 |
| **リスク管理ダッシュボード** 💡 | 承認モード運用率・却下率・高リスク変更の件数 | リスクを定量で経営に共有 | journal 集計の可視化 |
| **マルチクラウド価値の可視化** 💡 | AWS/GCP/Cloudflare をまたいで一元管理できる利点を示す | ベンダーロックイン回避の訴求 | プロバイダ横断のサマリビュー |
| **チーム生産性メトリクス** 🪝 | デプロイ頻度・リードタイム（DORA 的指標） | 開発組織の健全性を測る | run/GitHub イベントから算出 |
| **利用量メータリング** 💡 | （将来 SaaS 化するなら）run 数・コストの課金計測 | 製品化の土台 | run メトリクスを課金単位に |
| **セキュリティ保証の提示** 💡 | ガードレール・承認・監査の仕組みを顧客説明資料に | 受注・信頼獲得の材料 | §3 の機能群をドキュメント化 |

## 横断テーマ（複数目線にまたがる土台）

これらは単体機能というより、多くの機能の前提になる「効く順」の基盤:

1. **構造化 plan（terraform -json）** … コスト・ポリシー・サマリ・ドリフトすべての精度を底上げ。
2. **コスト基盤（Infracost）** … ユーザーの安心、管理者の予算ガード、ビジネスのレポートに直結。
3. **監査ログ + 認証/RBAC** … 管理者・ビジネスのガバナンス系がすべてここに乗る。
4. **メトリクス基盤（run 集計）** … 作者の改善ループ、ビジネスの ROI/DORA に直結。
5. **リモート state + 同時実行制御** … 個人ツールからチームツールへの分水嶺。

## 優先度の提案

| ティア | 候補 | 狙い |
|---|---|---|
| **クイックウィン**（既存フックに乗る） | plan の人間向けサマリ / 通知 / 構造化 plan / tflint・tfsec 必須化 / spec テンプレ追加 | 小コストで体験と安全性を底上げ |
| **中期**（新規モジュール） | コスト概算(Infracost) / 認証・RBAC / 監査ログ / リモート state / PR ベース GitOps | 個人ツール → チーム運用へ |
| **戦略**（製品化視点） | ポリシーガードレール / ドリフト検知 / コストダッシュボード / eval ハーネス / メータリング | ガバナンス・信頼性・ビジネス価値 |

## 次の一歩

- この中から **クイックウィン1〜2個**（例: 「plan の人間向けサマリ」＋「tfsec 必須化」）を選んで実装し、
  効果を見てから中期テーマに進むのが安全。
- 繰り返し効く作業手順（例: 「新プロバイダ対応の型」）が見えたら、ルート R-002 に沿って skill 化を検討する。
