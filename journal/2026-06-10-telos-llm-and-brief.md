# 2026-06-10 telos — pnpm 化・LLM 連携・Brief 追加・デザイン刷新

## ユーザーの指示（実装方針・要件）

1. パッケージ管理を **pnpm** にする
2. データの **インポート / エクスポート**機能をつける
3. **LLM API 接続をオプション機能**としてつける。LiteLLM での実装で良いが、**`claude -p` のようにサブスクリプションの利用もできる**ように
4. 画面デザインを**モダンなエンジニア感**にする
5. **ビジネス側に説明する用の資料作成**を行いたい。その資料のデザインは**ごちゃごちゃさせず見やすく**

## 設計判断とその理由

### pnpm ワークスペース化
- `app`（フロント）と新設の `server`（LLM ブリッジ）を1つの pnpm workspace にまとめた。理由: LLM 機能のために別パッケージ（サーバ）が必要になり、単一ディレクトリでは収まらなくなったため。

### LLM 連携を「ローカルブリッジ」方式にした理由（重要）
- 要件「`claude -p` でサブスクリプションを使う」は、**ブラウザから直接は実現できない**。CLI を叩くにはローカルプロセスが要る。よって `server/` に薄い中継サーバを置く構成は要件から必然。
- ブリッジは2形態を環境変数で切替: `claude-cli`（`claude -p` をサブプロセスで実行、サブスク利用・APIキー不要）と `litellm`（OpenAI 互換エンドポイントへ転送、APIキー）。
- **API キーをブラウザに持たせない**方針。キーはサーバ側に隔離。仕事情報を扱うアプリとして一貫。
- LLM は**任意の上乗せ**。OFF でも全機能がヒューリスティックで動く。AI は Goals（抽出・なぜ提案）と Brief（下書き生成）に接続。

### Brief（ビジネス説明資料）を4つ目のツールに
- 既存 Slides（報告）とは別物として独立させた。Brief は意思決定を仰ぐ一枚資料で、構成を「背景 → 提案 → 期待効果 → コスト → 意思決定のお願い」に固定（Minto のピラミッド原則に沿い、結論と判断依頼に絞る）。
- 「ごちゃごちゃさせない」要件 → **成果物は白い『紙』として描画**。アプリ本体のダーク UI とは意図的にコントラストさせ、届ける相手が読むものは余白とコントラストで見やすく。

### デザイン: ダークなエンジニア系 + 明るい成果物
- アプリ chrome: ダーク背景・mono フォントのアクセント・グラデーション・グリッド感。
- 成果物（Slides / Brief）: 白い紙。「道具はエンジニア向け、アウトプットはビジネス向け」という役割分担を視覚化。

- ブリッジを起動し health / 404 / 不正リクエスト、そして実際の `claude -p` 補完（"OK" を返す）まで疎通確認済み。
