# 2026-05-28 marginalia アイデア出し（初回）

iPhone 向けの PDF / EPUB リーダーアプリ。自分で出版するのではなく、ダウンロードした PDF / EPUB を「読みやすく表示し、メモを書く」ためのツール。Kindle への不満を解消する個人リーダーという位置づけ。

## 決定

### ディレクトリ名: `marginalia`
- 「余白の書き込み・欄外メモ」の意。
- 狙い: 本アプリの核である「メモを書ける」コンセプトを名前そのものが体現する。既存の `ariadne` 系コンセプト命名にも馴染む。
- 候補だった他案: lectern（書見台＝読書体験）, codex（写本）, folio（ページ）。

### コアコンセプト（採用）: 「紙の身体感覚を取り戻すデジタルリーダー」
- ユーザーが挙げた Kindle への不満（文字サイズ調整・進捗の分かりにくさ・学びの振り返り・パラパラ読み）は、いずれも「紙の本が持っていた身体的な感覚の喪失」に対する違和感、という一本の軸で説明できる。
- ユーザー本人が「紙の身体感覚を取り戻す」という軸に「しっくりくる」と同意。

### 最優先で解決したい不満: 進捗・位置の分かりにくさ
- ユーザーの言葉:「このくらいの厚みで今どのくらい進んでいる」「何がどこらへんに書いてあったか」「真ん中あたりだったな、と真ん中を開く」ができない。
- これは認知科学でいう **空間記憶（spatial memory）** の喪失。紙は内容を物理的な位置（厚み・左右ページ・上下）とセットで覚えられるが、デジタルはそれを奪う。進捗が分からないことと内容を思い出せないことは同じ根っこ。
- 言い換え: **本を「スクロールする文字列」ではなく「厚みと位置を持つ物体」として扱う。**

### 設計原則: ページは取り込み時に確定し、以後不変（文字サイズ問題の解決）
- ユーザー提案。「文字サイズ可変 vs 位置の安定」のジレンマを**時間軸で分離**して解く。
- **取り込み時に 1 回だけ**好みの文字サイズでレイアウトを確定し（=「このアプリにおけるページ」を物理的に決める）、**読書中は不変**にする。読書中は紙の本と同じでページが絶対に動かないので空間記憶が完全に効く。Kindle の「ページ概念が不安定」を治す発想。
- ユーザーの読書スタイル: 読み始めたら文字サイズは変えない。最初に読みやすいサイズを固定して読む。
- 含意:
  - **EPUB も取り込み時に 1 回リフロー→固定ページ化**。PDF と EPUB が「動かないページの集合」という**1 つの内部表現に正規化**され、扱い・実装・体験が一本化される。
  - 文字サイズは「読書中の機能」ではなく**「取り込み時の設定」**に降りる。読書 UI からサイズ変更が消えてシンプルに。
  - **付箋・メモはページ番号ではなく本文テキストのアンカーに紐付ける**（再ページネーションしてもズレない保険）。要合意。
- 残論点:
  - **再レイアウトできない PDF**（スキャン画像／図表・数式が緻密な論文）は再レイアウトせず**余白トリミング＋列フィット**で対応。PDF は中身で 2 系統（テキスト主体＝再レイアウト＆固定 / 画像・複雑＝原本維持＋トリミング）。原則「取り込み時確定・以後不変」は両系統共通。
  - 後からサイズ変更したい稀なケースは**再取り込み相当の重い操作**として許容する想定（要確認）。
- **MVP スコープ判断（合意済み）**: テキスト再レイアウトは実装難度が高いため MVP では行わない。MVP は PDF のみで、PDF は元々固定ページなので「位置の安定（空間記憶の前提）」は再レイアウトなしで満たせる。文字サイズは余白トリミング + ズームで対応し、テキスト再レイアウトは EPUB 対応と合わせフェーズ後半へ。実装計画は `marginalia/roadmap/phase1-mvp.md` に策定。

### MVP の柱（確定）: 「小口 + 付箋」
- B（厚みを掴んでジャンプ）と D（メモ・ハイライトを位置のピンで刺す）は別機能ではなく、**同じ「厚み」の上に統合された 1 つの中核 UI** として作る。これを MVP の中心に据える（ユーザー同意）。
- イメージ: 画面**右端に小口（本の側面＝厚み）を常時表示**。自分のハイライト／メモが本の中の位置に対応した高さに**色付き付箋**として刺さる。小口を掴んでドラッグでその位置にジャンプし、付箋のあたりを掴めば「あのメモの場所」に飛べる。= 付箋だらけの紙の本を付箋の出っ張りで開く体験の再現。
- 付箋は**メモの種類で色分け**（例: 重要 / 疑問 / 引用 / あとで読む）。

### 本棚レベルの進捗一覧（理想・フェーズ2候補）
- 登録した PDF/EPUB を「何をどこまで読んだか」一覧で把握したい（ユーザー要望）。MVP 範囲外かもしれないが理想。
- 厚みメタファーの棚スケール拡張として一貫: 本棚に並んだ各本の背表紙が既読分だけ満ちる。
- MVP で進捗データは必然的に持つため、本棚一覧は「そのデータの見せ方」を足すだけで地続き。

### フェーズ分け（暫定）
- **フェーズ1（MVP）**: 取り込み → 表示 → 右端の小口 → 色分け付箋 → 掴んでジャンプ → 進捗記録。1冊の読書体験を完成。
- **フェーズ2**: 本棚／ライブラリ画面で全書籍の進捗を背表紙メタファーで一覧。
- **フェーズ3**: 文字サイズ調整の高度化・学びノートの横断検索・パラパラ読み・エクスポート／復習・EPUB リフロー対応。

## 出したアイデア（進捗・空間記憶まわり）

- **A. 常時表示の厚みゲージ** — 既読を左の束・未読を右の束として物理的な厚みで見せる。位置 No. でも % でもなく束の感覚。
- **B. 厚みを掴んでジャンプ** — スクラブバーではなく束を直接掴んで飛ぶ。真ん中を掴めば真ん中へ（=「真ん中を開く」操作）。
- **C. 本全体の俯瞰マップ** — 全ページを帯／グリッドで一望し章を色分け。「あの図は後半の右上」を位置で想起。
- **D. メモ・ハイライトを位置のピンとして地図に刺す** — 進捗（不満2）と学びの振り返り（不満3）を一画面に統合。marginalia の名前と噛み合う。

（文字サイズ調整／パラパラ読み／学びノート等のアイデアも出したが、最優先ではないため詳細は次回深掘り）

## パラパラ読み × 触覚フィードバック（深掘り・採用）
- パラパラ読みは独立機能ではなく、**B（小口を掴んでジャンプ）と同じ小口ジェスチャーの速度連続体**として作る。ゆっくり＝精密ジャンプ、速く擦る＝サムネがパラパラ流れるスキミング、指を止めれば開く。
- **触覚フィードバック（Core Haptics / Taptic Engine）を組み合わせる**。「紙の身体感覚を取り戻す」コンセプトの文字通りの実現で、空間記憶を視覚だけでなく触覚にも拡張する。
  - ページが流れるごとの微細な tick（速度に応じて間隔・強度を変える）。
  - **付箋を通過する瞬間の触覚を種類ごとにパターンで変える（採用・白眉）**。画面を見ずとも指の感触だけで「あのへんに重要な付箋がある」が分かる＝紙の付箋の出っ張りを探り当てる体験の再現。視覚（色）＋触覚（パターン）で位置を多感覚的に記憶。
  - 章境界でスナップする触覚／始端・終端で壁の触覚。
- **紙めくりの効果音をオプションで搭載**（ユーザー要望）。触覚が主役、音は任意。
- 必須: 触覚オフ・効果音オフの設定。技術留意点として高速パラパラ時は haptic を間引く。

## 重要な設計トレードオフ（要検討）

- 空間記憶が効く前提は「位置が安定していること」。文字サイズを大きくするとレイアウトが変わり位置がズレて空間記憶が壊れる。Kindle はこれで失敗している。
- 方針案:
  - PDF（固定レイアウト）→ 空間記憶と相性抜群。厚み・位置がそのまま使える。
  - EPUB（リフロー）→ 絶対位置でなく相対位置（%）ベースで厚みを保つ工夫が要る。
- 「文字サイズの自由さ」と「位置の安定」の両立がこのアプリの技術的な肝。
