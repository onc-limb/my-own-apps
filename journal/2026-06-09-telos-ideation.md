# 2026-06-09 telos — ビジネス情報整理アプリの立ち上げ

## ユーザーの依頼

- 新しいディレクトリを作り、**ビジネスパーソンとして使えそうな情報整理 web アプリ**を **TypeScript** で作る
- 名前は「周りを見て」考える（リポジトリの命名慣習に合わせる）
- 欲しい機能（ユーザーの言葉のまま）:
  1. 新規でソフトウェアをリリースするときのチェックリスト
  2. プロジェクトの進捗を報告するスライド作成
  3. 会話の履歴から本当に叶えたい目的を見つけ出す

## 命名の判断

- 既存アプリの命名: ariadne（ギリシャ神話）、marginalia（ラテン語）、habiterm（合成語）— 含意のある一語名が慣習
- **telos（τέλος = 目的・最終目標）を採用**。理由: 第3機能「本当の目的を見つける」がアプリの軸であり、チェックリストもスライドも「目的に向かう道具」として1本のコンセプトに束ねられるため。ariadne と同じギリシャ語系で揃う。

## コンセプトの判断

- 3機能は一見バラバラだが、「情報が記録のままで、目的に向かう道具になっていない」という共通の根を持つと整理した
- コアコンセプト: **情報を「記録」で終わらせず、目的（telos）に向かう道具に変える**
- 各機能のエビデンス: チェックリスト = Haynes 2009（手術チェックリストで死亡率半減）/ スライド = Minto ピラミッド原則（結論ファースト）/ 目的発見 = 5 Whys（Ohno 1988）+ JTBD（Christensen 2016）。出典は telos/docs/concept.md に記録

## 技術スタックの判断

- **Vite + vanilla TypeScript、クライアントサイドのみ、localStorage 永続化**
- 理由: リポジトリの既存慣習（image-comp が vanilla Vite、investment が Vite + Solid）に合わせ、依存最小・メンテフリーを優先。仕事情報を扱うので外部送信しない設計が望ましい
- Goals の会話分析は MVP では LLM を使わず、マーカー検出（「〜したい」「〜が課題」等）+ 頻出語 + ガイド付き 5 Whys のヒューリスティックで実装。LLM 化は Phase 3 候補

## 宿題・次の方針

- localStorage 依存なのでデータのエクスポート / インポートが Phase 2 の最優先候補
- Goals の抽出精度はヒューリスティックの限界があるため、使ってみて不満が出たら LLM API 接続を検討
