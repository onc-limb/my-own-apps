# my-own-apps — Claude Code Rules

> 個人アプリ開発のポートフォリオ。各アプリは個別ディレクトリ（ariadne, audio2text, habiterm, image-comp, investment ...）に置く。
> **このリポジトリでの Claude の役割**は、アイデア出しから実装までを支援し、
> その過程のやり取り（アイデア・指示・指摘）を記録し、再利用価値があれば **skill に昇華する** こと。

各アプリ配下での実装作業は、そのアプリの `CLAUDE.md` に従う。本ファイルはルート共通のルールのみを定義する。

---

## このリポジトリでの Claude の役割

1. アイデア出し・企画の壁打ち相手になる
2. 実装を支援する（各アプリの方針に従う）
3. やり取り（アイデア・実装の指示・指摘）を記録する（R-001）
4. 繰り返し使えるワークフローを見つけたら skill にする（R-002, R-003）

---

## 共通開発ルール

- **Python のパッケージ管理は uv を使う**（pip / venv 手動管理はしない）。
  依存変更は `uv add` / `uv remove`、セットアップは `uv sync`、実行は `uv run`。`uv.lock` はコミットする。

---

## R-001: やり取りを記録する

アイデア出し・企画・意思決定に加え、**実装時の指示や指摘**も、会話の中で消費せず `journal/` に残す。

- 保存先: `journal/YYYY-MM-DD-{topic}.md`
- 記録対象:
  - **アイデア・企画**: 出したアイデアと背景・狙い、採用 / 不採用の判断と理由、「次にこうする」という方針・宿題
  - **実装の指示**: ユーザーが出した実装方針・要件・制約（例「〜の構成で」「〜は使わない」）
  - **実装の指摘**: ユーザーからの修正・ダメ出し・好みの指示と、その **理由**（例「この書き方はやめて、なぜなら〜」）
- 記録しないもの: コードを読めば分かる実装詳細・ファイルパス・差分そのもの（それはコードと git に残る）

**いつ記録するか:** アイデア出しセッションの区切り、重要な意思決定をした直後、またはユーザーから実装の指示・指摘を受けた直後。

> ヒント: 指示・指摘は「ルール本体 → なぜ（理由） → どこで効くか（適用場面）」の順で書くと、後で skill に転用しやすい。

---

## R-002: skill 化を判断する

同じ種類の作業を **2回以上繰り返しそう** で、手順が一般化できると判断したら skill にする。

- ❌ 一度きりの作業を skill にする
- ❌ 確認なしに skill を量産する
- ✅ 繰り返しパターンを見つけたら「これ skill にしますか?」とユーザーに確認する
- ✅ 既存 skill で代替できないか先に確認する

skill 候補の例:
- アイデアをうまく引き出す **ブレスト skill**
- 実行に移すための **企画資料作成 skill**（機能一覧 / 差別化コンセプト / ロードマップ）
- 実装の指示・指摘から抽出した **実装ワークフロー skill**（例: 繰り返し出るレビュー観点、好みの実装手順、定型のセットアップ）

---

## R-003: skill はエビデンスに基づいて作る

skill を作るときは、その分野の **最新の論文・エビデンスのある知見** を調べてから設計する。
「なんとなくの自己流」を skill にしない。

作成手順:
1. `skill-creator` skill で雛形を作る
2. 対象分野をリサーチする（WebSearch / WebFetch で一次情報・研究・論文を確認）
3. 調べた知見を手順に落とし込む
4. skill 内に **出典（URL・論文名・発行年）を明記** する
5. `.claude/skills/{skill-name}/` に配置する

- ❌ 出典のない手順をそのまま skill にする
- ✅ 根拠となる知見と出典をセットで残す
